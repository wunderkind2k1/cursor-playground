# Post 5 — Version Bump

**Wann:** Bei jedem Git-Release (v1.x.0)  
**Typ:** Playbook-Update  
**L-Stufe:** L1  
**Ton:** Editorial, sachlich — auch bei Updates kein Changelog-Sprech

---

## Post (Vorlage — copy-paste)

Ein öffentliches Protokoll ist nur so glaubwürdig wie sein Changelog.

Bei **Ideen auf Probe** liegt die Source of Truth auf GitHub — versioniert, nachvollziehbar, nicht nur als schöne Notion-Seite. Wenn sich das Curriculum ändert, soll erkennbar sein: *was*, *warum*, und *was ich daraus gelernt habe*.

Version **v[X.Y]** ist gerade live.

**Was sich geändert hat:**
→ [Bullet 1 — konkret, nicht technisch]  
→ [Bullet 2]  
→ [Bullet 3 — optional]

**Warum:** [Ein Satz — z. B. „Nach dem ersten Cycle" / „Launch-Material ergänzt" / „Gate G3 verschärft, weil …"]

Ich spiegele relevante Änderungen in die Notion-Lesefassung — GitHub bleibt die Referenz.

— Source of Truth: https://github.com/wunderkind2k1/cursor-playground/releases/tag/v[X.Y].0  
— Lesefassung: [Notion-URL]

Wenn du das Protokoll nutzt: Welche Stelle würdest du als Nächstes schärfen?

---

## Beispiel — v1.1.0 (bereits veröffentlicht)

Ein öffentliches Protokoll ist nur so glaubwürdig wie sein Changelog.

Version **v1.1** von **Ideen auf Probe** ist live. Der Kern des Playbooks war in v1.0 schon da — diese Version macht es launchfähig.

**Was sich geändert hat:**
→ Notion-Setup: öffentlicher Seitenbaum, privater Cycle-Workspace, Sync-Anleitung  
→ LinkedIn-Material: Posts, Profiltext, Launch-Reihenfolge  
→ Curriculum-Visual (ASCII) für Notion und Newsletter #0

**Warum:** Ein Protokoll, das nur im Repository existiert, validiert keine Idee — es muss lesbar und begleitbar sein.

— https://github.com/wunderkind2k1/cursor-playground  
— [Notion-URL]

**Was sich geändert hat:**

**Warum:** Der Launch-Text sollte das Protokoll ernst nehmen — nicht wie eine Produktankündigung klingen.

## Hook-Variante (erste Zeile A/B)

**A (Default):** *Ein öffentliches Protokoll ist nur so glaubwürdig wie sein Changelog.*

**B:** *Playbooks, die sich nie ändern, sind entweder fertig — oder nie ehrlich getestet worden.*

---

## Redaktionsnotizen

| Element | Warum |
|---------|-------|
| Changelog als Glaubwürdigkeit | Editorial-Rahmen statt „Update!!!" |
| „Was" + „Warum" | Seriosität, nicht nur Diff |
| GitHub vor Notion | Source-of-Truth-Prinzip wiederholen |
| Frage am Ende | Konsistent mit Post-Serie |
| Zwei ausgefüllte Beispiele | Autor kann copy-pasten |

**Länge Vorlage:** ~900 Zeichen (Updates dürfen kürzer sein als Launch-Posts)
